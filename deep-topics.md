## 1 · NGINX — the four jobs people lump into "a proxy"

NGINX shows up as **four separate containers**, each doing one job, so viewers never confuse them.

### 1a. Load Balancer (L7) — 🎬 Day 22
📄 `infra/nginx/lb.conf`
- `upstream` pool of `core-api-1` / `core-api-2`
- **`least_conn`** algorithm (vs default round-robin) — and *why* least-conn for uneven request cost
- **Health & ejection:** `max_fails=3 fail_timeout=10s` — how NGINX takes a sick replica out
- **Transparent failover:** `proxy_next_upstream error timeout http_502 http_503` — retry the next replica
- `X-Forwarded-For` so the origin still sees the real client
- ⚖️ L7 vs L4 · round-robin vs least-conn · DNS round-robin (no health awareness) · client-side LB

### 1b. API Gateway — 🎬 Day 17
📄 `infra/nginx/gateway.conf`
- **Path routing:** `/bff/` → BFF, `/api/` → core-api (prefix trim with `proxy_pass .../`)
- **The gateway ≠ the LB:** one guarded *door* + policy, not traffic spreading
- **SSE-safe proxying:** `proxy_buffering off` + `proxy_http_version 1.1` + `Connection ""` so events flush
- Not published to the host — only the edge can reach it (private network / "VPC")
- ⚖️ private gateway vs direct service exposure vs a full service mesh (Istio)

### 1c. CDN / caching reverse proxy — 🎬 Day 21
📄 `infra/nginx/cdn.conf`
- **`proxy_cache_path`** + `keys_zone` — a real on-disk cache
- **Immutable asset caching:** hashed filenames → `max-age=3600, immutable`
- **Stale-while-revalidate:** `proxy_cache_use_stale` + `proxy_cache_background_update on`
- **Cache stampede protection:** `proxy_cache_lock on`
- **Observability:** `X-CDN-Cache $upstream_cache_status` (HIT/MISS on camera)
- ⚖️ CDN vs origin serving · cache-busting via content hash vs query string

### 1d. TLS termination — 🎬 Day 16
📄 `infra/nginx/tls.conf` · `infra/nginx/gen-certs.sh`
- Terminate HTTPS **once** at the edge boundary, forward plain HTTP inside the private net
- `ssl_protocols TLSv1.2 TLSv1.3`; local dev certs generated by script
- **HTTP→HTTPS 301 redirect** server block; `X-Forwarded-Proto https`
- ⚖️ terminate at edge vs at origin vs full mTLS everywhere

## 2 · kind + Kubernetes — local orchestration, the real objects

### 2a. The cluster (kind) — 🎬 Day 24
⚠️ **Gap to fill before Day 24:** there's no `kind-config.yaml` yet — the manifests exist but the
cluster-bootstrap file doesn't. We need: a `kind` cluster config (control-plane + workers, and the
**`extraPortMappings`** that expose the edge NodePort `30888` to the host), plus a `make load-images`
step (`kind load docker-image worldcup-core-api:latest …`) because kind doesn't pull local images.
- ⚖️ Docker Compose only (no self-heal/autoscale) vs kind vs Swarm/Nomad · why kind for teaching

### 2b. Deployments & Services — 🎬 Day 24
📄 `infra/k8s/00-namespace.yaml` · `20-core-api.yaml` · `30-services.yaml`
- A `Deployment` per service; **the Service *is* the in-cluster load balancer** (contrast with NGINX LB)
- `imagePullPolicy: IfNotPresent` (pairs with `kind load`)
- **NodePort** edge (`nodePort: 30888`) as the public entry — note an Ingress/LoadBalancer in real clusters

### 2c. Scaling, self-healing, rollout — 🎬 Day 25
📄 `infra/k8s/20-core-api.yaml`
- **HPA** (`autoscaling/v2`): `minReplicas 2 → maxReplicas 6`, target **CPU 70%**
- **Probes:** `readinessProbe` (take out of rotation) vs `livenessProbe` (restart) on `/health`
- **Resource `requests`/`limits`** — and why the HPA needs `requests` to compute utilization
- Rolling updates & self-healing (kill a pod on camera, watch it come back)
- ⚖️ manual scaling vs HPA · rolling vs recreate · readiness vs liveness (the #1 confusion)

### 2d. Stateful data tier + replication — 🎬 Day 26
📄 `infra/k8s/10-data.yaml`
- Postgres **streaming replication**: primary `POSTGRESQL_REPLICATION_MODE: master` →
  replica `mode: slave` + `POSTGRESQL_MASTER_HOST`
- core-api split: `DATABASE_URL` (primary, writes) vs `READ_DATABASE_URL` (replica, reads)
- Redis readiness via `redis-cli ping`
- ⚖️ single DB vs read replicas · **replication lag / read-after-write staleness** · sync vs async

## 5 · BFF + external data integration — 🎬 Days 9–10
📄 `services/bff/src/` (`index.ts`, `footballData.ts`, `espn.ts`, `apiFootball.ts`)
- Per-frontend **aggregation & shaping** (only what each panel needs)
- **BFF as the anti-corruption layer / integration point** — the browser never calls a 3rd-party API;
  the BFF does, server-side, so keys stay secret and responses are cached & mapped to `@wc/types`
- **Real upstream fan-out (3 providers):**
  - `footballData.ts` → **football-data.org v4** (`X-Auth-Token`, ~10/min) — 104 fixtures, scores, standings
  - `espn.ts` → **ESPN Site API** (`fifa.world`, **keyless**) — commentary, lineups+formations, stats → SSE
  - `apiFootball.ts` → **API-Football v3** (`x-apisports-key`, 100/day, optional) — played-match detail
- **Cross-provider id resolution:** football-data fixture → ESPN event by date + team name
- **Resilience:** in-memory TTL cache that **serves stale on upstream error** (never throws to a route)
- **Mode switch:** `USE_REAL = FOOTBALL_API_KEY set && DATA_SOURCE !== "sim"`
- ⚖️ BFF vs frontend→core directly · one-API-for-all · GraphQL gateway · calling 3rd-party APIs from the browser (keys leak, CORS, no caching)

## 5b · Simulator data adapters (ports/adapters) — 🎬 Day 4
📄 `services/simulator/app/adapters/` (`simulated.py`, `real_api.py`, `thesportsdb.py`)
- One `fetch_live_states()` port, three adapters chosen by `DATA_SOURCE`:
  `sim` (deterministic, offline) · `real` (football-data.org) · `thesportsdb` (TheSportsDB, keyless `123`)
- ⚖️ simulated vs real source · why an abstraction boundary lets you swap feeds with zero downstream change

## 6 · Real-time — three transports, one event — 🎬 Days 11–15
📄 `services/simulator` · `services/bff/src` (SSE) · scoreboard (WS) · `services/edge` · Redis
- **Polling** (Standings, ~15s, cached/SWR) — simple, wasteful, laggy
- **SSE** (Match Center) — one-way, auto-reconnect, HTTP-native; `/bff/sse/match/:id`
- **WebSocket** (Scoreboard) — bidirectional, lowest latency, more ops cost
- **Redis pub/sub fan-out**: simulator publishes once → all subscribers receive → push per transport
- Sticky sessions & scaling stateful connections
- ⚖️ one transport everywhere (the #1 mistake) · in-process vs broker fan-out · long-polling · WebRTC
